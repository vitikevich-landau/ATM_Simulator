01

Как устроен поток данных

Главный инвариант проекта: состояние течёт строго в одну сторону — писатель → снимки-копии → рендер. Рендер физически не может испортить движок, и на этом держится чистота под ThreadSanitizer. Долгое «обслуживание» — это сон на condition_variable с отпущенным локом, поэтому чтение снимка никогда не ждёт задержку обслуживания.

FrameDiffer → терминал

В терминал уходят только изменившиеся строки. Полный кадр в legacy conhost — 42–50 мс; дифф — ~1 мс.

С чего начать

базовый

Деньги — целые копейки, не double

core/Money.hpp:18

Money = std::int64_t, сумма хранится в минорных единицах (50000 = 500.00).

Правило: деньги, проценты, количество — целые или decimal; double только для физики и графики. Классика бага новичка — хранить деньги в double и «лечить» сравнением с epsilon.
базовый

enum class + switch без default

core/Types.cpp:5

Все перечисления домена — scoped enum: нет неявного int, имена не текут. switch по всем значениям без default:.

Зачем без default: при добавлении нового значения компилятор (-Wswitch//W4) напомнит про забытый case. default: «съел» бы это предупреждение.
базовый

const-корректность интерфейса

core/Cashbox.hpp

Наблюдатели (balance(), can*()) — const; мутаторы — нет. Предикаты и изменения читаются раздельно.

На вырост: без const на геттере его не позвать из потока-читателя, которому отдали const&. const по умолчанию — и документация, и защита.
базовый

ANSI-коды в одном месте

console/Terminal.hpp:114

Все escape-последовательности — в namespace ansi как inline-функции. Никаких «магических» \033[ по коду.

Инкапсуляция констант: захотелось truecolor — правишь одно место. Каждый окрашенный участок обязан закрываться reset(), иначе цвет «течёт» дальше.
базовый

Не храни то, что можешь вычислить

engine/ServiceStages.hpp:30

Этап обслуживания («вводит PIN», «отсчёт купюр») не хранится, а вычисляется чистой функцией serviceStageAt(op, progress).

Принцип: хранимое производное состояние надо синхронизировать и защищать от гонок; вычисляемое — нет. Поэтому фича не тронула тонкий цикл ожидания.

Снимки-DTO + mutable mutex

engine/Snapshots.hpp:3

Читатели никогда не получают ссылок на живое состояние — только копии, снятые под коротким локом. Мьютекс mutable в const-методах.

Strategy + Factory + unique_ptr

engine/ServiceTimeProvider.hpp:17

Время обслуживания — сменное распределение за общим интерфейсом. Три реализации спрятаны в анонимном namespace; наружу — только фабрика.

Эталонная связка: Strategy (поведение) + Factory (создание по конфигу) + unique_ptr (владение полиморфом). virtual ~T() = default обязателен при удалении через базу.
средний

RAII для потоков

engine/ScopedEngineThreads.hpp

Деструктор joinable-потока зовёт std::terminate. Обёртка гарантирует stop()+join() при любом выходе, идемпотентно.

Порядок важен: сначала requestStop() — сигнал на выход, потом join(), иначе join повиснет на спящем потоке. По сути — ручная замена std::jthread.

Дифф-рендеринг кадров

Изюминки проекта

Места, ради которых этот проект интересно читать: тонкая синхронизация, корректность на границах, детерминизм. Разбирать их лучше с комментариями в коде — они там подробные.

продвинутый

cv без потери пробуждений — atomic не спасает

engine/AtmEngine.cpp:535

Смена условия делается под тем же мьютексом, на котором ждут, затем notify_all(). Иначе возможна потеря пробуждения в зазор между проверкой предиката и засыпанием.

Главный миф студента: «переменная же atomic, зачем мьютекс?» Атомарность защищает от гонки данных, но не от потерянного пробуждения — это гонка логики вокруг cv.
продвинутый

Прерываемое «слайсовое» ожидание

engine/AtmEngine.cpp:172

Вместо sleep_for(длительность) ожидание нарезано на слайсы: реагирует на pause/stop и на истечение терпения в очереди; лок во сне отпущен.

Тонкость: wait_for здесь без предиката — легально только потому, что обёрнут внешним for(;;), который каждый раз перечитывает режим. Не копируй «голый» wait_for.
продвинутый

Прогресс через параметры формулы

engine/AtmEngine.hpp:199

Спящий поток не может обновлять «процент». Движок публикует тройку (план / накоплено / старт слайса), а snapshot() довычисляет прогресс в момент чтения.

Идея в копилку: если производитель не может часто обновлять значение — публикуй параметры формулы, вычисляй у читателя. Пауза (слайс закрыт) честно замораживает шкалу.
продвинутый

Проверка переполнения до сложения

core/Cashbox.cpp:15

balance_ <= max - amount, а не balance_ + amount > max: переполнение знакового целого — UB, то есть сама проверка была бы сломана.

Запомни паттерн: для a + b <= MAX пиши a <= MAX - b (при b > 0). Тесты бьют ровно в границу: столько — можно, на единицу больше — нельзя.
продвинутый

Транзакционность: verify-then-commit

core/Operation.cpp:16

Сначала все проверки, потом обе мутации счёта и кассы. Невозможно состояние «счёт уменьшили, а касса не выдала» — на этом стоит property-тест сохранения денег.

Заметь: applyOperation — свободная функция, а не метод. Логика, связывающая два объекта, не принадлежит ни одному из них.
продвинутый

Детерминизм: два RNG, splitmix64

engine/AtmEngine.hpp:172 · scene/ActorAnim.cpp:22

У mt19937_64 нет синхронизации — один движок на два потока это гонка. Держат два независимых. В сцене вместо RNG — хэш splitmix64(id, окно).

Hash-based randomness: одна пара (id, момент) всегда даёт одну позу → golden-тесты не флапают, состояние хранить не нужно. Разные аспекты — разные соли, иначе скрытая корреляция.
продвинутый

Валидация как защита preconditions <random>

config/ConfigLoader.cpp:94

stddev > 0, probability ∈ [0,1], isfinite, потолок count — их нарушение это UB без диагностики. Ловят на границе системы, а не в глубине движка.

Урок: у <random> есть preconditions, нарушение которых — тихий UB. Каждая проверка снабжена комментарием «зачем» со ссылкой на потребителя значения — подражай этому стилю.
продвинутый

Однопоточный эталон перед потоками

engine/SimulationRunner.cpp

Та же доменная логика без потоков и реального времени: дискретно-событийная модель, время — double, макс. очередь считается sweep-line.

Методология важнее кода: сначала отладь логику детерминированно и однопоточно, потом добавляй потоки — иначе отлаживаешь гонки и доменные баги одновременно.

06

Маршрут изучения

Порядок, в котором проект открывается без боли — от чистого ядра к многопоточности и рендеру.

Доменное ядро без потоков

core/: Money, Account, Cashbox, applyOperation. Здесь — типобезопасность денег, инварианты, verify-then-commit. Ни одной строки многопоточности.

Однопоточный эталон

engine/SimulationRunner.cpp: та же СМО-логика на числах и без потоков. Понять модель, прежде чем ловить гонки.

Многопоточный движок

engine/AtmEngine.{hpp,cpp} + Snapshots.hpp: producer-consumer, протокол остановки, слайсовое ожидание, снимки-развязка. Читать вместе с комментариями.

Рендер и сцена

console/: дифф-рендер, абстракция терминала, MVP-конвейер сцены, reconciliation актёров, UTF-8 на трёх уровнях.

Инфраструктура

CMakeLists.txt, tests/simple_test.hpp, scripts/: слоистая сборка, самодельный тест-харнес, санитайзеры (ASan+UBSan вместе, TSan — отдельно).

Что подкрутили по итогам анализа производительности

  • 🔴 главное — operations() теперь копирует только хвост журнала под мьютексом, а не весь растущий log_ (было O(всего журнала) на каждом опросе ленты).
  • 🟡 — рендер кэширует табличную часть кадра и пересобирает только анимируемую сцену (раньше весь кадр строился на каждый кадр, и дифф выбрасывал ~73%).
  • 🟢 — единый fullSnapshot() под одним локом вместо пяти; std::erase_if вместо пересборки очереди; пауза ждёт событие, а не поллит; мелкие reserve.

Разбор и правки сделаны в ветке perf/part1-optimizations (влита в main). Изменения поведение-сохраняющие: прогон воспроизводим бит-в-бит, все 154 теста + ThreadSanitizer + AddressSanitizer зелёные. Осознанно не тронуты flush логгера (гарантия при падении) и пересоздание normal_distribution (сохраняет воспроизводимость).
