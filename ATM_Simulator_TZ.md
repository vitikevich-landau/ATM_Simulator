### О документе

ТЗ описывает консольное приложение на C++, моделирующее работу одного банкомата с очередью клиентов, случайным временем обслуживания и **неблокирующим** интерфейсом мониторинга/управления для администратора системы.

Помимо функциональных требований документ фиксирует архитектурные решения — модель параллелизма, структуру данных, паттерны проектирования. Без этого «консольная программа с очередью» рискует превратиться в один линейный цикл без реальной конкурентности, а именно конкурентность здесь — ключевой технический вызов задачи, а не побочная деталь.

Структура разделов частично опирается на привычную для ТЗ логику (в духе ГОСТ 19.201), но адаптирована под специфику разработки ПО: добавлены разделы про архитектуру, паттерны и тестирование, без которых документ был бы неполным для реальной разработки.

Решения, помеченные как «рекомендация», разработчик может менять по своему усмотрению. Решения без такой пометки обязательны.

## 1. Термины и определения

| Термин | Определение |
|---|---|
| Банкомат (АТМ) | Симулируемое устройство, обслуживающее одного клиента за раз |
| Клиент | Виртуальный «посетитель» банкомата: номер, запрошенная операция, сумма, терпение |
| Очередь | Список клиентов, ожидающих обслуживания (по умолчанию — FIFO) |
| Операция | Банковское действие: проверка баланса, снятие, внесение и т.д. |
| Касса банкомата | Текущий остаток наличности внутри банкомата |
| Терпение (patience) | Максимальное время, которое клиент готов ждать в очереди, прежде чем уйти |
| Уход из очереди (reneging) | Клиент покидает очередь, не дождавшись обслуживания |
| Отказ от очереди (balking) | Клиент видит слишком длинную очередь и сразу решает не вставать в неё (опциональное расширение) |
| Администратор | Пользователь консоли управления, наблюдающий и управляющий системой |
| Техническое обслуживание (ТО) | Режим, в котором банкомат временно не обслуживает клиентов (инкассация и т.п.) |
| СМО | Система массового обслуживания — теоретическая модель, к которой относится задача (одноканальная СМО с очередью и уходом клиентов) |

## 2. Назначение и цели проекта

Цель — разработать консольное C++-приложение, моделирующее одноканальную систему массового обслуживания (один банкомат — одна очередь), в которой:

- администратор может **параллельно, не блокируя работающую симуляцию**, смотреть отчёты и отдавать команды управления;
- параметры симуляции (интенсивность прихода клиентов, распределение времени обслуживания, вероятности операций и т.д.) задаются конфигурацией, а не зашиты в код;
- банкомат может уходить на техническое обслуживание с реалистичным поведением очереди — часть клиентов уходит, часть остаётся ждать.

Проект одновременно закрывает две задачи:

1. практическую — попробовать многопоточность, синхронизацию и паттерны проектирования на живом, не игрушечном примере;
2. концептуальную — реализовать классическую задачу теории массового обслуживания (single-server queue with reneging) и получить из неё осмысленную статистику.

## 3. Общее описание работы системы

Система состоит из трёх независимых по своей природе процессов, которые исторически принято реализовывать раздельно:

1. **Процесс прихода клиентов** — генерирует новых клиентов и ставит их в очередь. Не зависит от того, как быстро работает банкомат.
2. **Процесс обслуживания** — банкомат забирает из очереди по одному клиенту, «выполняет» операцию со случайной длительностью, обновляет счёт клиента и кассу банкомата.
3. **Процесс управления** — администратор в любой момент читает состояние системы или отдаёт команду (пауза, остановка, техобслуживание), не дожидаясь, пока банкомат «освободится».

Именно то, что эти три процесса идут одновременно и не должны мешать друг другу — и есть суть задачи. Технически это означает минимум два потока выполнения (симуляция и консоль), а не один цикл с проверкой `cin` между итерациями — такой цикл не сможет реагировать на команду администратора, пока не закончит обслуживать текущего клиента.

## 4. Функциональные требования

### 4.1. Клиенты и очередь

- Каждый клиент должен получать уникальный порядковый номер (`ClientId`) в момент появления в системе.
- Система должна поддерживать два режима генерации клиентов, выбираемых конфигурацией (`arrival_mode`):
  - **batch** — заранее сгенерированный конечный список из N клиентов;
  - **poisson** — непрерывный поток с экспоненциально распределёнными интервалами между приходами (классическая модель прихода в СМО), параметр — среднее число клиентов в минуту.
- У каждого клиента при создании должны генерироваться (согласно конфигурации): тип операции (по весам вероятностей), сумма операции (для снятия/внесения — случайно в заданном диапазоне), терпение — максимальное время ожидания в очереди.
- Очередь — FIFO по умолчанию. Приоритетная дисциплина обслуживания — не входит в обязательный объём, но архитектура не должна этому мешать (см. раздел 15).
- Если терпение клиента истекло, а его очередь ещё не подошла — клиент должен покинуть очередь (`LeftQueue`), и это должно попадать в отчётность так же, как обслуженный клиент, просто с другим статусом.
- *Опционально:* если очередь длиннее заданного порога, новый клиент с заданной вероятностью может не вставать в неё вообще (balking) — предлагаю как расширение, а не обязательное требование, см. раздел 14.

### 4.2. Операции банкомата и учёт денежных средств

| Операция | Обязательность | Баланс клиента | Касса банкомата |
|---|---|---|---|
| Проверка баланса (CheckBalance) | Обязательно | не меняется | не меняется |
| Снятие наличных (Withdraw) | Обязательно | уменьшается | уменьшается |
| Внесение наличных (Deposit) | Обязательно | увеличивается | увеличивается |
| Мини-выписка (MiniStatement) | Расширение | не меняется | не меняется |
| Перевод между счетами (Transfer) | Расширение | меняются два счёта | не меняется |
| Смена ПИН-кода (ChangePin) | Расширение | не меняется | не меняется |

Правила:

- Снятие должно проверяться дважды: хватает ли средств на счёте клиента **и** хватает ли наличности в кассе банкомата. При нехватке любого из двух — операция завершается неуспешно с понятной причиной, это не аварийная ситуация, а штатный исход (см. раздел 6.5).
- **Денежные суммы должны храниться в целочисленных минимальных единицах валюты** (копейки/центы), а не в `float`/`double` — иначе после нескольких сотен операций накопится ошибка округления. Это принципиальное требование для любой финансовой логики, даже игрушечной.
- Внесение наличных увеличивает и счёт клиента, и кассу банкомата, а снятие уменьшает обе величины на одну и ту же сумму — деньги не берутся из ниоткуда и не пропадают в никуда. Отсюда точный инвариант для тестов (см. раздел 11): величина **(сумма всех счетов − касса банкомата)** не меняется никогда; эквивалентно — и сумма счетов, и касса по отдельности меняются ровно на «успешные внесения минус успешные снятия». *(Уточнено на M1: ранее здесь стояла «сумма счетов + касса», но при внесении обе величины растут, поэтому их сумма меняется на удвоенную величину — сохраняется именно разность.)*

### 4.3. Мониторинг и администрирование (неблокирующее)

Администратор должен иметь возможность в любой момент, не останавливая и не замедляя обслуживание клиентов:

- увидеть, какой клиент (по номеру) обслуживается прямо сейчас и какую операцию он выполняет;
- увидеть, сколько клиентов всего обслужено с начала работы;
- увидеть список всех клиентов, находящихся в очереди сейчас, и сколько их;
- получить отчёт по конкретному клиенту: статус, история операций, текущий баланс счёта;
- посмотреть баланс счёта конкретного клиента;
- получить отчёт о состоянии банкомата: остаток наличности, статус, аптайм;
- посмотреть журнал операций (кто, что, когда сделал), с фильтрами по клиенту / типу операции / количеству последних записей.

Требование к отзывчивости: чтение отчётов не должно ждать завершения текущей операции обслуживания — ни на секунды, ни на миллисекунды больше, чем нужно для копирования небольшого снимка данных.

Помимо разовых команд-запросов система должна поддерживать **непрерывный режим наблюдения** (живая визуализация процесса) — см. §4.8. Live-режим строится на тех же snapshot-методах и подчиняется тому же требованию неблокирующего чтения.

### 4.4. Отчётность и журналирование

- Каждая выполненная операция должна фиксироваться в журнале: `id клиента`, `тип операции`, `сумма`, `результат` (успех/причина отказа), `баланс после операции`, `таймстамп`.
- Журнал должен быть доступен для чтения администратором во время работы системы (см. 4.3) и экспортируем в файл (в v1 — CSV; JSON отложён в v2) по команде — как для последующего анализа, так и на случай сохранения результатов после завершения симуляции.
- При завершении работы система должна печатать/сохранять сводную статистику: всего обслужено, всего ушло по терпению, среднее время ожидания, среднее время обслуживания, максимальная длина очереди, коэффициент загрузки банкомата (ρ = λ/μ — отношение интенсивности прихода к интенсивности обслуживания, стандартная метрика СМО).

### 4.5. Режим технического обслуживания

Сценарий (по аналогии с приездом инкассаторов):

1. Администратор подаёт команду `maintenance start [длительность]`.
2. Банкомат перестаёт принимать в обслуживание новых клиентов. Клиент, который обслуживался в этот момент, по умолчанию **доводится до конца** (реалистичнее, чем обрывать операцию на середине) — прерывание на середине можно сделать конфигурируемой альтернативой, см. раздел 16.
3. Каждый клиент, стоящий в очереди, должен «решить», ждать ли окончания ТО: с заданной конфигурацией вероятностью клиент либо уходит сразу (`LeftQueue`), либо остаётся ждать — но не дольше своего терпения, даже если ТО не закончилось.
4. Пока идёт ТО, новые клиенты по умолчанию продолжают прибывать и вставать в очередь (в реальной жизни это так и есть — люди подходят и видят, что банкомат недоступен); это тоже можно вынести в конфигурацию.
5. ТО завершается либо по истечении заданной длительности, либо явной командой `maintenance stop`. После этого банкомат возвращается в обычный режим и продолжает обслуживать оставшихся в очереди клиентов в порядке FIFO.

### 4.6. Управление жизненным циклом банкомата

- `pause` / `resume` — приостановить и возобновить обслуживание (очередь продолжает расти во время паузы, в отличие от ТО, где приход клиентов настраивается отдельно).
- `stop` — плавная остановка: доработать/прервать текущую операцию (по той же логике, что и ТО), сохранить журнал и статистику, корректно завершить потоки, завершить процесс.
- `restart` / `again` — прервать текущий прогон (не дожидаясь его завершения), остановить потоки, **перечитать конфиг с диска** (правки файла применяются без перезапуска программы) и запустить прогон заново с начала.
- Обработка `SIGINT`/`SIGTERM` должна приводить к такой же плавной остановке, а не к аварийному завершению процесса.

### 4.7. Конфигурирование системы

Все параметры симуляции должны задаваться конфигурационным файлом (рекомендация — JSON), без пересборки программы:

- параметры прихода клиентов (режим, интенсивность, диапазоны сумм, диапазон терпения, веса вероятностей операций);
- распределение времени обслуживания и его параметры (см. 6.3);
- начальная касса банкомата, начальные балансы клиентов, валюта (код и, опционально, формат вывода — символ, группировка разрядов, десятичный знак; см. §9);
- параметры поведения при ТО (вероятность ухода, длительность по умолчанию);
- seed генератора случайных чисел — для воспроизводимости тестовых прогонов;
- `time_scale` — коэффициент ускорения модельного времени относительно реального (например, 1 модельная минута = 1 секунда реального времени) — полезно, чтобы не ждать полчаса ради проверки сценария с ТО.

Пример конфигурации — раздел 9.

### 4.8. Живая визуализация процесса (Live-режим)

> **Статус пункта:** добавлено при согласовании (06.07.2026). Причина — в §4.3/§8 интерфейс администратора построен на модели «запрос → снимок» (pull) и не даёт непрерывной картины процесса: между командами обратные таймеры терпения, рост очереди, убыль кассы и смена обслуживаемого клиента происходят внутри Engine, но администратор их не видит, пока снова не наберёт команду. Раздел вводит **опциональный** режим непрерывного наблюдения. Командный интерфейс §8 остаётся обязательным и самодостаточным.

Помимо разовых команд «запрос-ответ» (§4.3) администратор должен иметь возможность **непрерывно наблюдать** за работой банкомата в реальном времени: очередь, текущий обслуживаемый клиент, касса и ключевые счётчики обновляются на экране сами, без повторного ввода `status`/`queue`. Это отдельный режим отображения консоли, а не отдельный процесс симуляции.

**Назначение.** Наглядная «панель диспетчера» одноканальной СМО: видно, как клиенты прибывают, ждут, уходят по терпению (§4.1) или при ТО (§4.5), как меняется загрузка банкомата и уровень кассы. Двойная ценность — демонстрационная и как инструмент отладки конкурентности.

**Принципиальное архитектурное ограничение (обязательно).** Live-режим — это **только читатель**. Он строится **исключительно** на существующих snapshot-методах `AtmEngine` (§6.3) и **не имеет** права мутировать очередь, кассу, счета или состояние банкомата. Тем самым сохраняется модель «один писатель — много читателей» (§6.1): рендерер добавляется как ещё один читатель через `shared_lock`/атомарные снимки и не создаёт новых путей записи. Рендер обязан работать по копиям-снимкам (DTO из §6.3, намеренно без `mutex`/`atomic` внутри), а не по «живым» ссылкам на внутреннее состояние Engine.

#### 4.8.1. Два режима работы консоли

**Решение (согласовано):** по умолчанию приложение стартует **в live-режиме** (`ui.live_mode: true`, §9). Управление в v1 — только текстовыми командами §8 (горячие клавиши отложены, см. §4.8.6).

| Режим | Что делает | Как включается |
|---|---|---|
| **live** (по умолчанию) | Экран периодически перерисовывается снимком состояния (dashboard). Ввод команд §8 доступен в строке ввода внизу экрана. | Стартовый режим (`ui.live_mode: true`); на лету — команда `live` |
| **command** | Классический REPL из §8: ввёл команду — получил ответ, экран не перерисовывается сам. | Команда `live off`; либо `ui.live_mode: false` |

> **Примечание v1.** CLI-флаги `--live`/`--no-live` не реализованы: единственный
> позиционный аргумент — путь к JSON-конфигу (`atm_sim [config.json]`). Режим
> задаётся ключом `ui.live_mode` и переключается командами `live` / `live off`.

Live-режим — надстройка над командным интерфейсом, а не замена: приложение должно оставаться полностью управляемым и в command-режиме, и при недоступном TTY (деградация, §4.8.7).

```
│   #129  CheckBalance   ждёт 00:12   терпение 04:48   ████████░░  ok        │
│   #130  Deposit 150.00 ждёт 00:47   терпение 03:13   ██████░░░░  ok        │
│   #131  Withdraw 80.00 ждёт 01:58   терпение 00:22   ██░░░░░░░░  !LOW      │
├───────────────────────────────────────────────────────────────────────────┤
│ Обслужено: 127   Ушли (терпение): 14   Ушли (ТО): 2   Макс. очередь: 7    │
├─ последние события ───────────────────────────────────────────────────────┤

```

Зоны экрана и их источники данных:

| Зона | Источник (snapshot-метод) | Обязательность |
|---|---|---|
| Шапка: состояние, касса, аптайм, ρ, режим/частота кадров | `snapshot()` + агрегаты `stats()` | Обязательно |
| Текущий клиент + прогресс операции | `snapshot().currentClientId` + `clientReport(id)` | Обязательно |
| Очередь с временем ожидания и **остатком терпения** | `queueSnapshot()` (+ поле `remainingPatience`, §4.8.3) | Обязательно |
| Строка агрегированных счётчиков: **обслужено `N / T`** (T — общий план прогона `clients.count`), ушли, макс. очередь | `snapshot()` / `stats()` | Обязательно |
| Лента последних событий | `operations({last: N})` | Рекомендация |
| Индикатор режима/частоты `[LIVE ▮ N fps]` и возраст кадра | локально в рендерере | Рекомендация |
| Командная строка ввода внизу | локальный буфер ввода | Обязательно (см. §4.8.5) |

**Полоса прогресса обслуживания** отображает долю прошедшего модельного времени от ожидаемой длительности операции (индикатор, а не точный прогноз — фактическая длительность задаётся `ServiceTimeProvider`, §6.3). **Полоса остатка терпения** рассчитывается по `waitingTime` и `patience`; при остатке < ~10% строка помечается `!LOW` — визуально видно, кто вот-вот уйдёт по reneging (§4.1).

**Форматирование сумм и счётчик клиентов.** Касса, балансы и суммы операций выводятся под валюту `atm.currency` — символ и его позиция, группировка разрядов, десятичный знак (§9); в плотных колонках (очередь, лента событий) суммы даются компактно, без символа валюты. Счётчик «Обслужено `N / T`» показывает прогресс относительно общего плана прогона (`clients.count`) — во время прогона видно, близко ли конец (иначе общее число клиентов видно лишь на стартовом баннере, скрытом альтернативным буфером).

#### 4.8.3. Единый согласованный кадр и данные для визуализации (обязательно)

- **Один кадр = один согласованный снимок.** Собирать экран из нескольких независимых вызовов (`snapshot()`, `queueSnapshot()`, `operations()`) в разные моменты нельзя — получится «рваная» картинка (очередь из одного мига, касса из другого, счётчик скакнул назад). Требуется композитный read-метод (рекомендуемо `AtmEngine::dashboardSnapshot()`) под **одним** захватом `shared_lock`, возвращающий единый DTO на весь экран.
- **Расширение DTO под визуализацию.** Snapshot-контракты §6.3 нужно дополнить производными скалярами, посчитанными **внутри** снимка (не на стороне рендера из «живых» `time_point`):
  - `ClientSnapshot`: `remainingPatience` (остаток терпения);
  - кадр/`AtmSnapshot`: флаг `lowCash` (сравнение с `low_cash_threshold` из §9), раздельные счётчики `renegedByPatience` и `renegedByMaintenance`, прогресс текущей операции `serviceProgress`, при активном ТО — `maintenanceEtaSeconds`.
- **(Рекомендация)** для живого тренда загрузки/длины очереди — потокобезопасный кольцевой буфер временного ряда фиксированной глубины (спарклайн внутри консоли); при переполнении старые точки вытесняются без роста памяти.

#### 4.8.4. Частота обновления и неблокирующий рендер (обязательно)

- Частота перерисовки задаётся конфигурацией (`refresh_hz`, §9), рекомендуемо 2–10 Гц, по умолчанию 4. Согласуется с `time_scale` (§4.7): при сильном ускорении модельного времени частоту имеет смысл повышать.
- Рендерер за кадр делает набор snapshot-вызовов (лучше — один `dashboardSnapshot()`), **копирует DTO, отпускает блокировки, и только затем** форматирует и печатает строки. Форматирование и вывод в терминал **не должны** выполняться под `shared_lock` — иначе рендер начнёт замедлять поток обслуживания (нарушение §5 и §6.2 — та же ошибка, что `sleep_for`, но на уровне вывода).
- Рендер выполняется в **отдельном потоке** (`std::jthread`, RAII — §6.4), управляемом `stop_token`, а не в потоке Engine и не встраиванием задержки в цикл обслуживания.
- Без мерцания: перерисовка через home-курсор + очистку до конца экрана, без полного `clear` на каждый кадр; при неизменившемся снимке кадр можно не перерисовывать (дифф).
- **Health интерфейса.** Дашборд показывает возраст последнего успешного кадра и метку `stale`, если снимок не обновлялся дольше ~2× периода — чтобы отличать «система остановилась» от «рендер не успевает».

#### 4.8.5. Совмещение живого вывода и ввода команд

Ключевая сложность: экран сам перерисовывается, но администратор при этом вводит команды §8.

**Модель ввода v1 (портируемая, без raw-режима).** Поскольку в v1 управление — только полнотекстовыми командами с Enter (не одиночными клавишами), ввод реализуется стандартным `std::getline` в потоке `AdminConsole`, без перевода терминала в raw/termios-режим. Это делает ввод одинаковым на Linux и Windows без платформенного кода. Dashboard занимает верхнюю область экрана, нижняя строка — ввод; render-поток перерисовывает **только область над строкой ввода** и восстанавливает позицию курсора (ANSI `ESC 7`/`ESC 8`). Известное ограничение: терминал в cooked-режиме сам отражает набираемые символы, поэтому перерисовка кадра во время активного набора может дать кратковременный артефакт в строке ввода (снимается на следующем кадре либо паузой перерисовки, пока строка ввода непуста). Полноценный raw-режим с ручным эхом — в v2 вместе с горячими клавишами.

- В live-режиме **все команды §8 остаются доступными** — это надстройка над вводом, а не замена.
- Строка ввода (нижняя зона) не должна затираться перерисовкой: рендерер печатает dashboard над строкой ввода и восстанавливает позицию курсора и содержимое редактируемой строки после каждого кадра.
- Ввод не блокирует рендер и наоборот. Рекомендация: чтение stdin — в потоке `AdminConsole` (§6.3), рендер — в отдельном потоке; **вывод в терминал — единственный разделяемый ресурс двух потоков** и сериализуется одним механизмом (`std::osyncstream` C++20 либо явный `mutex` вокруг записи в stdout), чтобы кадр и эхо ввода не перемешивались.
- Результат разовой команды (например, `client 128`, `stats`), выполненной внутри live-режима — **рекомендация**: либо приостановить перерисовку и показать полноэкранный ответ до нажатия клавиши, либо вывести в отдельную прокручиваемую/overlay-зону. Недопустимо только затирание ответа следующим кадром до того, как администратор его увидел.

#### 4.8.6. Горячие клавиши (v2, отложено)

> **Статус (согласовано):** в v1 **не реализуются** — управление только текстовыми командами §8. Горячие клавиши требуют raw-режима терминала (termios / Win32 console mode) и переносятся в v2. Таблица ниже — целевой ориентир для v2.

В live-режиме, помимо полноценного ввода команд, допустимы горячие клавиши для частых действий — как ускорение, не замена команд §8:

| Клавиша | Действие | Эквивалент §8 |
|---|---|---|
| `q` / `Esc` | выйти из live в command-режим | — |
| `F2` / `F3` | пауза / возобновление | `pause` / `resume` |
| `F4` / `F5` | старт / стоп техобслуживания | `maintenance start` / `stop` |
| `/` или `Tab` | фокус в командную строку для полнотекстовых команд | — |
| `+` / `-` | сменить частоту кадров / прокрутка ленты | — |
| `F8` / `Ctrl+C` | плавная остановка (как §4.6) | `stop` |

Горячие клавиши обязаны использовать **те же** потокобезопасные методы Engine (`requestPause()`, `requestMaintenance()` и т.д. из §6.3), а не собственные пути воздействия на состояние. Никакая клавиша не даёт live-режиму прав записи в обход §4.8.

#### 4.8.7. Деградация и переносимость

- Если stdout **не** интерактивный терминал (перенаправление в файл/пайп/CI) или терминал не поддерживает управляющие последовательности — live-режим автоматически **отключается** (даже при `ui.live_mode: true`), приложение работает в обычном command-режиме. Обязательно: наличие/отсутствие визуализации **не влияет** на корректность и результаты симуляции (§5, детерминированность при фиксированном seed). Уровни: (1) TTY+ANSI → полный дашборд; (2) бедный/узкий терминал → монохром без альт-буфера, периодическая перепечатка компактного снимка; (3) не-TTY → плоские построчные дампы без ANSI (пригодны для `tee`/`grep`); (4) `ui.live_mode: false` / `live off` → чистый pull-режим §8.
- Детектирование — через `isatty(STDOUT_FILENO)` (и `TERM`). Управление терминалом (курсор, очистка, неканонический ввод) — платформенно-зависимо: Linux — ANSI + termios (обязательно, §5); Windows — VT-режим консоли (по остаточному принципу, §16 п.4).
- Логику рендеринга держать изолированной за интерфейсом (`LiveRenderer`/`DashboardView`, модуль `console/`, §12), чтобы платформенные детали и выбор «рисовать/не рисовать» не протекали в Engine (Strategy, §6.4).

#### 4.8.8. Взаимодействие с другими режимами

- **Пауза (`pause`)**: dashboard продолжает обновляться, состояние `Paused`, очередь визуально растёт (§4.6).
- **Техобслуживание (`maintenance`)**: шапка показывает `Maintenance` и остаток времени ТО (`maintenanceEtaSeconds`); уходящие клиенты помечаются в ленте как `LeftQueue (ТО)` (§4.5).
- **Остановка (`stop`/SIGINT)**: перед завершением потоков рендерер обязан вернуть терминал в нормальное состояние (показать курсор, сбросить режим, восстановить канонический ввод) через **RAII-обёртку** — чтобы терминал не «завис» в альтернативном режиме даже при аварийном завершении.

## 5. Нефункциональные требования

- **Неблокирующее взаимодействие.** Команды чтения (отчёты) должны выполняться и возвращать результат за конечное малое время независимо от фазы обслуживания клиента.
- **Оперативность управляющих команд.** `pause`/`stop`/`maintenance` обязаны вступать в силу быстро — реализация не имеет права «слепо» ждать окончания текущей случайной задержки обслуживания (см. 6.2 — это не тривиально и требует конкретного решения, а не просто `sleep_for`).
- **Потокобезопасность.** Любая структура данных, доступная более чем из одного потока, должна быть либо защищена синхронизацией, либо атомарна, либо передаваться только через потокобезопасные копии-снимки.
- **Детерминированность для тестов.** При фиксированном seed и конфигурации повторный прогон должен давать идентичный результат — обязательное условие для регрессионных тестов.
- **Расширяемость.** Добавление нового типа операции, нового распределения времени обслуживания или новой команды администратора не должно требовать правок в неродственных модулях.
- **Логирование.** Минимум два независимых потока информации: бизнес-журнал операций (для отчётов) и технический лог уровня DEBUG/INFO/WARN/ERROR (для разработчика).
- **Кроссплатформенность (согласовано).** Целевые платформы — **Windows 10+ и Linux** (обе поддерживаются). Приоритет разработки — Windows, но платформенно-специфичный код изолируется в тонком слое `TerminalBackend`: включение VT-режима консоли на Windows (`ENABLE_VIRTUAL_TERMINAL_PROCESSING`) vs ANSI на Linux, определение ширины окна, `isatty`. За счёт отказа от raw-режима в v1 (§4.8.5) объём платформенного кода минимален. Примечание: ThreadSanitizer доступен на Linux/clang, поэтому TSan-гейт (§11) прогоняется в Linux-сборке (CI или WSL), даже если основная разработка идёт на Windows.
- Специальных требований к производительности, кроме отсутствия блокировок интерфейса администратора, не предъявляется — это учебный/демонстрационный симулятор, а не высоконагруженная система.
- **Визуализация не снижает отзывчивости.** Живая визуализация (§4.8) не имеет права снижать отзывчивость обслуживания: рендер работает по копиям-снимкам, форматирование и вывод в терминал выполняются вне блокировок состояния Engine. Наличие, отсутствие или частота live-режима **не влияют** на детерминированность прогонов при фиксированном seed.

## 6. Архитектура и модель параллелизма

### 6.1. Общая схема потоков

```
┌───────────────────────────────┐        ┌────────────────────────────────┐
│   Главный поток (main)         │        │   Поток симуляции (Engine)      │
│                                 │        │                                  │
│   AdminConsole                 │        │   AtmEngine::run()               │
│   читает команды из stdin,     │─ команды ─▶  цикл: взять клиента из     │
│   парсит, вызывает методы       │◀ снимки данных ─│  очереди → случайная       │
│   AtmEngine, печатает результат │        │  задержка (interruptible) →      │
└───────────────────────────────┘        │  применить операцию → лог        │
                                          └────────────────────────────────┘
                    │                                      │
                    ▼                                      ▼
        ┌──────────────────────────────────────────────────────────┐
        │   Общее состояние: очередь, касса, счета, журнал, статус  │
        │   (shared_mutex + atomic + condition_variable_any)         │
        └──────────────────────────────────────────────────────────┘
                                          ▲
                                          │
                          ┌───────────────────────────────┐
                          │  Поток генерации прихода        │
                          │  (можно объединить с Engine,     │
                          │  но раздельно — честнее к модели) │
                          └───────────────────────────────┘
```

Ключевой архитектурный принцип, который сильно упрощает всё остальное: **очередью, кассой и счетами мутирует исключительно поток `AtmEngine`** (единственный «writer»). Поток консоли администратора только читает (через snapshot-методы) либо просит сменить режим работы (pause/stop/maintenance) через потокобезопасные примитивы. Модель «один писатель — много читателей» резко снижает риск гонок данных по сравнению с «все пишут во всё».

### 6.2. Синхронизация и почему нельзя просто `sleep_for`

Если время обслуживания клиента моделировать через `std::this_thread::sleep_for(долгая_случайная_задержка)`, то команда `stop`, отправленная администратором, применится только **после** окончания этой задержки — то есть система не будет неблокирующей на самом деле, просто блокировка переедет на уровень ниже.

Правильное решение — ждать не безусловно, а по предикату:

```cpp
// Вместо этого:
std::this_thread::sleep_for(delay);

// Делать так — поток просыпается ЛИБО когда истекло время,
// ЛИБО когда изменился флаг состояния (пауза/стоп/ТО),
// в зависимости от того, что наступит раньше:
{
    std::shared_lock lock(stateMutex_);
    wakeUp_.wait_for(lock, delay, [this] {
        return state_.load() != AtmState::Serving;
    });
}
```

В C++20 у `std::condition_variable_any` есть перегрузки, принимающие `std::stop_token` — это ровно то, что нужно для потока, запущенного через `std::jthread`: поток просыпается либо по предикату, либо по внешнему запросу остановки, без ручной возни с дополнительным атомарным флагом «пора выходить».

> **Примечание из практики реализации (Windows/MSVC).** В ряде версий MSVC STL связка `std::condition_variable_any` + `std::shared_mutex` + `std::stop_token`/`std::jthread` падает в рантайме («unlock of unowned mutex», порча стека) — как в перегрузках со `stop_token`, так и без них. Поэтому фактическая реализация ядра следует явно разрешённому §12 fallback'у: `std::jthread`/`stop_token` заменены на `std::thread`, а `shared_mutex` + `condition_variable_any` — на обычные `std::mutex` + `std::condition_variable`. Остановка — через `requestStop()`, который под мьютексом состояния выставляет `state_ = Stopped` и вызывает `notify_all()`; предикаты всех ожиданий проверяют `state_ == Stopped`. Это канонический паттерн condition_variable без потери пробуждений (изменение состояния и его проверка упорядочены общим мьютексом). Вызывающий (main, тесты) ОБЯЗАН вызвать `requestStop()` перед `join()`. Мгновенность применения `pause`/`stop` (§5, §14) сохранена. Модель «один писатель — много читателей» (§6.1) остаётся логически; на масштабе симулятора сериализация читателей на обычном мьютексе незаметна.

`std::jthread` (C++20) сам хранит `stop_source`, сам передаёт `stop_token` в функцию потока и сам вызывает `request_stop()` + `join()` в деструкторе — на нём `main()` получается заметно короче и без риска забыть `join`.

### 6.3. Ключевые модули (иллюстративные интерфейсы)

Ниже — черновые сигнатуры, фиксирующие контракты между модулями. Реализацию (приватные поля, детали) разработчик волен менять.

```cpp
using Money = std::int64_t;   // минимальные единицы валюты (копейки/центы), НЕ float/double; вывод человеку — formatMoney(...) под валюту (§9)

enum class OperationType   { CheckBalance, Withdraw, Deposit /*, MiniStatement, Transfer, ChangePin — расширения */ };
enum class ClientState     { Waiting, InService, Served, LeftQueue /*, Balked — расширение */ };
enum class AtmState        { Idle, Serving, Paused, Maintenance, Stopped };
enum class OperationStatus { Success, InsufficientFunds, InsufficientCash, InvalidAmount, Overflow };

struct OperationOutcome {
    OperationStatus status;
    Money balanceAfter;
};

class Account {
public:
    explicit Account(AccountId id, Money initialBalance);
    Money balance() const;
    OperationOutcome withdraw(Money amount);
    OperationOutcome deposit(Money amount);
private:
    Money balance_;
    mutable std::mutex mutex_;
};

class Cashbox {
public:
    explicit Cashbox(Money initialAmount);
    Money balance() const;                 // atomic-чтение, не требует захвата shared_mutex целиком
    bool canDispense(Money amount) const;
    void dispense(Money amount);           // вызывается только из потока Engine
    void accept(Money amount);
private:
    std::atomic<Money> balance_;
};

// Strategy — время обслуживания как подключаемая стратегия
class ServiceTimeProvider {
public:
    virtual ~ServiceTimeProvider() = default;
    virtual std::chrono::milliseconds next(OperationType type) = 0;
};
class UniformServiceTime     : public ServiceTimeProvider { /* std::uniform_real_distribution */ };
class NormalServiceTime      : public ServiceTimeProvider { /* std::normal_distribution, усечение снизу нуля */ };
class ExponentialServiceTime : public ServiceTimeProvider { /* std::exponential_distribution */ };

// DTO для отчётов — намеренно НЕ содержат mutex/atomic внутри,
// чтобы их можно было спокойно копировать и отдавать наружу
struct ClientSnapshot {
    ClientId id;
    ClientState state;
    OperationType requestedOperation;
    std::chrono::seconds waitingTime;
};
struct AtmSnapshot {
    AtmState state;
    Money cashboxBalance;
    std::optional<ClientId> currentClientId;
    std::uint64_t totalServed;
    std::size_t queueLength;
};

class AtmEngine {
public:
    explicit AtmEngine(Config config);

    void run(std::stop_token stopToken);   // основной цикл, выполняется в отдельном потоке

    // --- потокобезопасные запросы для отчётов, вызываются из потока консоли ---
    AtmSnapshot snapshot() const;
    std::vector<ClientSnapshot> queueSnapshot() const;
    std::optional<ClientReport> clientReport(ClientId id) const;
    std::vector<OperationRecord> operations(const OperationFilter& filter) const;

    // --- команды управления, потокобезопасные, не блокируют вызывающего ---
    void requestPause();
    void requestResume();
    void requestStop();
    void requestMaintenance(std::chrono::seconds duration);
    void endMaintenance();

private:
    mutable std::shared_mutex stateMutex_;      // читатели не блокируют друг друга
    std::atomic<AtmState> state_{AtmState::Idle};
    std::condition_variable_any wakeUp_;

    ThreadSafeDeque<Client> queue_;
    Cashbox cashbox_;
    std::unique_ptr<ServiceTimeProvider> serviceTimeProvider_;
    OperationLog log_;
    std::atomic<std::uint64_t> totalServed_{0};
};

class AdminConsole {
public:
    explicit AdminConsole(AtmEngine& engine);
    void run();   // читает строку из stdin → парсит → выполняет → печатает результат
private:
    AtmEngine& engine_;
    CommandParser parser_;
};
```

Набросок `main()`, показывающий, как это связывается:

```cpp
int main(int argc, char** argv) {
    Config config = ConfigLoader::loadFromFile(argv[1]);
    AtmEngine engine(config);

    std::jthread engineThread([&engine](std::stop_token st) { engine.run(st); });
    // jthread сам пошлёт stop_request и сделает join в деструкторе —
    // не нужно вручную помнить об этом в конце main()

    AdminConsole console(engine);
    console.run();   // блокирует главный поток до команды stop/exit

    return 0;
}
```

Отдельное предостережение: если генератор случайных чисел (`std::mt19937_64` и т.п.) используется больше чем из одного потока (например, поток прихода клиентов и поток обслуживания используют один и тот же движок), это гонка данных — у `std::mt19937` нет внутренней синхронизации. Решение: либо свой экземпляр движка на поток, либо явный `mutex` вокруг общего.

### 6.4. Рекомендуемые паттерны проектирования

| Паттерн | Где применяется |
|---|---|
| State | Состояния банкомата (Idle/Serving/Paused/Maintenance/Stopped) и переходы между ними |
| Strategy | Подключаемые распределения времени обслуживания и модели прихода клиентов |
| Command | Команды администратора как объекты — упрощает добавление новых команд и, при желании, логирование/историю команд |
| Observer | Оповещение о событиях (клиент обслужен, банкомат остановлен) — удобно для журналирования и будущего расширения (например, метрики) без изменения ядра |
| Factory | Создание клиентов/операций по конфигурации |
| RAII | Управление потоками (`std::jthread`), файлами логов — ресурс освобождается в деструкторе, а не вручную |

### 6.5. Обработка ошибок

Три разных класса ошибок требуют разного обращения — смешивать их в одну модель (например, всё через исключения) обычно только усложняет код:

- **Программные/конфигурационные ошибки** (битый JSON, файл не найден) — исключения (`std::runtime_error` или свои типы), ловятся в `main()`, логируются, приложение корректно завершается.
- **Бизнес-исход операции** (не хватило денег на счёте или в кассе) — это НЕ исключение, а ожидаемый, частый результат бизнес-логики. Возвращается явно через `OperationOutcome`/`OperationStatus`. Кидать исключение на «недостаточно средств» — как кидать исключение на `false` в `if` — концептуально неверно.
- **Ошибки ввода администратора** (опечатка в команде) — выводится подсказка, симуляция не прерывается и не падает.

```cpp
struct Client {
    ClientId id;

    OperationId id;

    OperationType type;
```

Состояния клиента и банкомата:

| `ClientState` | Значение |
|---|---|
| Waiting | стоит в очереди |
| InService | обслуживается прямо сейчас |
| Served | обслуживание успешно завершено |
| LeftQueue | ушёл, не дождавшись (терпение истекло или началось ТО) |

| `AtmState` | Значение |
|---|---|
| Idle | работает, очередь пуста |
| Serving | работает и обслуживает клиентов |
| Paused | обслуживание приостановлено, очередь продолжает копиться |
| Maintenance | идёт техническое обслуживание |
| Stopped | система остановлена (терминальное состояние) |

| Команда | Описание |
|---|---|

`clients.count` — сколько всего клиентов сгенерировать за прогон;
`clients.initial_balance` — стартовый баланс счёта каждого клиента;
`clients.walk_seconds` — время подхода к банкомату (взятый из очереди клиент
сначала идёт к терминалу, потом начинается обслуживание, §4.1).

Блок `ui` — параметры живой визуализации (§4.8): `live_mode` — включать ли
live-режим при старте (в v1 CLI-флага нет; переключение на лету — команды `live`
/ `live off`); `refresh_hz` — частота перерисовки dashboard (рекомендуемо
2–10 Гц); `events_tail` — сколько последних событий показывать в ленте;
`show_progress_bars` — рисовать ли полосы прогресса/терпения; `color` —
использовать ли ANSI-цвета (при не-TTY игнорируется, см. §4.8.7). Ключи
анимированной сцены: `scene` — показывать ли сцену над таблицей; `scene_fps` —
частота кадров сцены (5..30); `scene_rows` — высота полосы (10..14);
`scene_effects` — спецэффекты (спиннер связи, купюры, «пар» злости). Подробнее —
[docs/scene_architecture.md](docs/scene_architecture.md).

Суммы указаны в минимальных единицах валюты (сотые доли), т.е. `50000000` = `500 000.00`. При выводе человеку сумма форматируется под валюту `atm.currency`: символ и его позиция, группировка разрядов и десятичный знак берутся из встроенной таблицы по коду валюты (EUR → `500 000.00 €`, USD → `$500,000.00`, RUB → `500 000.00 ₽`, JPY → `¥500,000`, неизвестный код → сам код как подпись). Любое из этих правил переопределяется необязательным блоком `atm.currency_format`:

```json
"atm": {
  "currency": "EUR",
  "currency_format": {
    "symbol": "€",                 // "" — печатать сам код валюты
    "symbol_position": "after",    // "before" ($100) или "after" (100 €)
    "space_between": true,         // пробел между символом и числом
    "group_separator": " ",        // "" — без группировки разрядов
    "decimal_separator": ".",
    "decimals": 2                  // 0..6
  }
}
```

В CSV-экспорт журнала (§10) и технический лог суммы пишутся «сырым» форматом без символа валюты — там важен стабильный машинный вид.

Технический лог: `logging.level` — порог важности (`debug` | `info` | `warn` | `error`; принимается также написание `warning`). Неизвестное значение не роняет запуск: программа предупреждает в stderr и использует `info`.

## 10. Логирование, статистика и экспорт данных

- Технический лог (DEBUG/INFO/WARN/ERROR) — для разработчика, отдельно от бизнес-журнала операций.
- Бизнес-журнал операций — то, что видит администратор через `operations`/`export`.
- Итоговая статистика при остановке: всего обслужено, всего ушло по терпению, среднее время ожидания и обслуживания, максимальная длина очереди, коэффициент загрузки банкомата ρ = λ/μ.
- Экспорт в CSV — для последующего анализа вне программы (например, построить график загрузки). JSON-экспорт отложён в v2.

## 11. Требования к тестированию

**Модульные тесты** — минимум перечисленного ниже. *Реализация:* используется
минимальный собственный тест-харнес [`tests/simple_test.hpp`](tests/simple_test.hpp)
(макрос `TEST(...)`, свой `main`), а не GoogleTest/Catch2 — внешний фреймворк
остаётся возможной заменой. Фактически 154 теста в 18 файлах покрывают в т.ч.
рендер, диффер кадров и анимированную сцену. Минимум:

- `Account` — списание/пополнение, отказ при недостаточном балансе;
- `Cashbox` — выдача/приём наличности, отказ при недостаточном остатке;
- реализации `ServiceTimeProvider` — с фиксированным seed проверить, что значения попадают в ожидаемый диапазон/распределение;
- переходы состояний банкомата — недопустимые переходы должны быть невозможны или явно обработаны;
- `CommandParser` — корректный разбор всех команд из раздела 8, включая некорректный ввод.

**Сценарные тесты:**

- «счастливый путь»: N клиентов проходят очередь полностью, отчёты и счётчики корректны;
- уход по терпению: клиент с коротким терпением уходит, не дождавшись;
- техобслуживание: при фиксированном seed заданная доля клиентов уходит, оставшиеся обслуживаются после возобновления работы — результат воспроизводим и сравним с ожидаемым;
- `pause`/`stop` применяются быстро — тест именно на время реакции, а не только на факт применения (см. 6.2).

**Проверка конкурентности:**

- сборка и прогон с `-fsanitize=thread` (TSan) и `-fsanitize=undefined,address` для выявления гонок данных;
- инвариант сохранения денег как property-based тест: величина (сумма всех счетов − касса банкомата) неизменна на всей серии операций (эквивалентно: и сумма счетов, и касса по отдельности меняются на «успешные внесения − успешные снятия»). Реализовано в тесте `money_conservation_invariant` (M1).

## 12. Технологический стек и структура репозитория

- **Стандарт:** C++20 (`jthread`, `stop_token`); при ограничениях тулчейна — минимум C++17 с ручной заменой `jthread`/`stop_token` на `thread` + `atomic<bool>`.
- **Сборка:** CMake ≥ 3.20.
- **Библиотеки.** Фактически: `nlohmann/json` — вендорен как single-header в
  [`third_party/nlohmann/json.hpp`](third_party/nlohmann/json.hpp) (без FetchContent);
  логирование — собственный `Logger` (не `spdlog`); тесты — собственный харнес
  `tests/simple_test.hpp` (не GoogleTest/Catch2); разбор аргументов — вручную
  (без CLI11). Рекомендации выше остаются возможными заменами при необходимости.
- **Терминал/визуализация (§4.8):** вывод в консоль сериализуется через `std::osyncstream` (C++20) либо явный `mutex`; управление терминалом — ANSI-последовательности + `termios` (Linux, обязательно), VT-режим консоли (Windows, опционально). Рендеринг dashboard изолирован в модуле `console/` за интерфейсом `LiveRenderer`/`DashboardView`; внешних TUI-библиотек не требует (при желании — опционально FTXUI/ncurses).
- **Инструменты качества:** `clang-format` (единый стиль), `clang-tidy` (статический анализ), Doxygen-комментарии для публичных интерфейсов.
- **Соглашение:** идентификаторы кода — на английском (стандартная практика даже в русскоязычных командах), комментарии и документация — на русском.

## 13. Этапы разработки

1. Доменное ядро без потоков: `Account`, `Cashbox`, `Client`, `Operation`, загрузка конфигурации, юнит-тесты.
2. Однопоточная симуляция очереди (без консоли администратора, вывод в stdout).
3. Вынесение симуляции в отдельный поток (`jthread`) + базовая консоль администратора (`status`, `queue`, `stop`).
4. Полная отчётность (`client`, `balance`, `operations`, `atm`, `stats`) и журнал операций.
5. Режим технического обслуживания и логика «уйти/остаться ждать».
6. **Live-режим визуализации (§4.8):** `LiveRenderer` поверх уже готовых snapshot-методов (опирается на этап 4), композитный снимок под одним локом, отдельный поток рендера, совмещение с вводом команд, деградация при неинтерактивном stdout. Этап только читает и ничего не мутирует. *(Горячие клавиши §4.8.6 отложены в v2.)*
7. Логирование в файл, экспорт отчётов (CSV), статистика СМО, полировка, документация, сборка с санитайзерами.

**Надстройки сверх исходного объёма (реализованы, документированы в [docs/](docs/)):**

8. **Этап обслуживания на дашборде** (`ServiceStages`): тематические стадии операции (карта → PIN → … → возврат карты) и прогресс в %; в `status` — строка «Сейчас».
9. **Анимированная сцена v1** (`console/scene/`): банкомат и очередь ASCII-человечков над таблицей, `ScenePresenter`/`SceneComposer`/`SceneCanvas`, дифф-рендер `FrameDiffer`, команда `scene [on|off]`. Рендер по-прежнему только читает движок (§4.8).
10. **Сцена v2:** фаза подхода к банкомату (`clients.walk_seconds`) в движке, личный темп ходьбы, болтовня соседей по очереди, сплеш-«самотест» при старте (`SplashScreen`).
11. **Оптимизации производительности:** единый `fullSnapshot()` под одним локом, кэш кадра, разделение частот снимок/кадр (влито в main).

Этот порядок не случаен: он специально устроен так, чтобы сначала отладить доменную логику без параллелизма, и только потом добавлять многопоточность поверх уже проверенного ядра — отлаживать бизнес-логику и гонки данных одновременно намного тяжелее.

## 14. Критерии приёмки

- Очередь корректно обрабатывается по FIFO с учётом ухода клиентов по терпению.
- Административная консоль отвечает на команды чтения за миллисекунды независимо от состояния очереди.
- `pause`/`stop`/`maintenance` вступают в силу без ожидания окончания текущей случайной задержки обслуживания.
- Отчёт по клиенту и по банкомату содержит корректные данные, сверяемые с журналом операций.
- Инвариант сохранения денег выполняется на протяжении всей серии операций.
- При техобслуживании доля ушедших/оставшихся клиентов соответствует заданной вероятности (статистически, на большом числе клиентов).
- Юнит-тесты покрывают ключевую бизнес-логику; прогон с ThreadSanitizer не выявляет гонок данных.
- **Live-режим (§4.8):** dashboard обновляется с заданной частотой и отражает актуальное состояние очереди/кассы/текущего клиента, при этом отзывчивость команд сохраняется (миллисекунды, независимо от идущей перерисовки).
- **Live-режим:** прогон с ThreadSanitizer при активном рендере не выявляет гонок данных — рендерер подтверждённо работает только как читатель (обращается лишь к snapshot-методам, не к внутренним контейнерам).
- **Live-режим:** при перенаправлении stdout в файл/пайп приложение корректно работает в command-режиме без ANSI-мусора, а итоговая статистика и журнал при фиксированном seed идентичны прогону без live.

## 15. Возможные направления развития (вне текущего объёма)

- Несколько банкоматов одновременно, общий менеджер (архитектура из раздела 6 этому не мешает — `AtmEngine` уже спроектирован как независимый инстанс).
- Balking — отказ вставать в слишком длинную очередь.
- Учёт кассы по номиналам купюр, а не общей суммой.
- Персистентность состояния между запусками (сохранить очередь/кассу в файл и восстановить).
- Веб- или GUI-обёртка поверх того же `AtmEngine` — консоль в этом случае всего лишь один из «клиентов» ядра, а не единственный интерфейс.

## 16. Открытые вопросы к уточнению

Поскольку внешнего заказчика в этом проекте нет — эти пункты стоит просто явно решить самому до начала кодирования, чтобы не переделывать архитектуру на середине пути:

1. Прерывать текущую операцию клиента при старте ТО или всегда доводить до конца? (В ТЗ по умолчанию — доводить до конца.)
2. Нужна ли персистентность состояния между запусками программы, или один прогон = одна сессия?
3. ✅ **РЕШЕНО.** Одна активная валюта на прогон (`atm.currency`), но формат вывода (символ и его позиция, группировка разрядов, десятичный знак) выбирается по коду валюты из встроенной таблицы (EUR/USD/RUB/GBP/JPY + разумный fallback для неизвестного кода) и переопределяется блоком `atm.currency_format` (§9).
4. Целевая ОС для сборки — только Linux, или сразу кроссплатформенно?
5. Нужен ли отдельный конфиг времени обслуживания на каждый тип операции (снятие обычно дольше проверки баланса) или один общий на все операции для первой версии?
6. Один банкомат в текущей версии, но сразу проектировать с расчётом на несколько — или отложить до раздела 15?
7. ✅ **РЕШЕНО.** Live-режим включён по умолчанию (`ui.live_mode: true`); стартуем сразу в dashboard, выход в command-режим — `live off`.
8. Частота перерисовки dashboard: фиксированная из конфига (`refresh_hz`) или адаптивная (выше при большом `time_scale`, ниже при пустой очереди)? (По умолчанию — фиксированная. **Открыто**, но не блокирует — начинаем с фиксированной.)
9. Как показывать результат разовой команды (`client <id>`, `stats`), выполненной внутри live-режима: полноэкранным ответом с паузой рендера до нажатия клавиши или в отдельной прокручиваемой зоне? (См. §4.8.5 — оставлено на выбор разработчика. **Открыто**, решается на этапе M6.)
10. ✅ **РЕШЕНО.** В v1 горячих клавиш нет — управление только текстовыми командами §8. Горячие клавиши (§4.8.6) — v2.
11. ✅ **РЕШЕНО.** Цвет включён по умолчанию: минимальная семантическая ANSI-палитра (красный — низкая касса / уход по терпению, жёлтый — ТО/пауза, зелёный — норма). Работает на Windows 10+ (VT) и в Linux одним кодом. Автоматический откат в монохром — вне TTY или при `ui.color: false`. Цвет через один флаг — так проще, чем поддерживать два отдельных рендерера.
12. ✅ **РЕШЕНО.** Обе платформы — Windows 10+ и Linux — через тонкий слой `TerminalBackend` (см. §5). Приоритет разработки — Windows.
